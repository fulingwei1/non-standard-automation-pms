# Base Platform Governance Plan

> 生成时间：2026-03-27
> 目标：对非标自动化项目管理系统进行“收敛式底座治理”，而不是推倒重构

---

## 一、结论先行

### 不建议做的事情
- 推倒式重构
- 全仓重写
- 大规模目录/框架迁移
- 一次性统一全部状态、全部接口、全部模型

### 必须开始做的事情
# 收敛式底座治理

原因很直接：
- 系统已具备可持续开发能力
- 关键业务线（合同→项目、资源冲突、权限系统、角色运营中心）已经成型
- 现在最大的风险不是“底座不能用”，而是“底座逐渐分叉、长胖、难维护”

所以正确策略不是拆楼重建，而是：

> **统一核心语义、统一关键入口、统一前后端契约。**

---

## 二、当前底座问题摘要

### 1. 项目状态模型分裂
已发现至少两套状态表达并存：
- 专用状态枚举：`ST01-ST30`
- 通用字符串状态：`active / archived / ...`

风险：
- 前端可能绕过专用状态流转
- 审计日志与状态更新链路不一致
- 不同页面/接口对同一状态的理解不一致

### 2. 模板领域模型不统一
模板已经开始成为系统高价值入口，但当前仍容易被当作“一个页面能力”，而不是稳定领域对象。

风险：
- 预览口径不统一
- 模板创建项目链路分散
- 模板结构（阶段/里程碑/任务/角色/周期）缺乏统一语义层

### 3. 销售→项目转换入口需统一
合同→项目创建已经成型，但如果继续在多个地方各自加入口，后续会出现：
- 多套创建规则
- 多套映射逻辑
- 多套审计记录

### 4. 资源冲突数据口径需收敛
资源冲突现在已经很强，但越强越怕多口径：
- 哪些资源算冲突
- 冲突时间窗口怎么算
- 冲突状态如何处理
- 处理后是否留痕

### 5. 前后端契约不一致风险仍在
当前项目管理模块虽已提升，但仍存在：
- 某些页面半打通
- 某些 API 已有，但前端未统一接入
- 个别功能通过“绕过式兼容”打通，不够规范

---

## 三、治理目标

这轮底座治理不追求“大而全”，而追求三件事：

## 目标 1：统一核心语义
至少统一：
- 项目状态
- 模板结构
- 销售转项目领域事件
- 资源冲突计算口径
- 审计事件含义

## 目标 2：统一关键入口
至少统一：
- 项目状态更新入口
- 模板创建项目入口
- 销售/合同转项目入口
- 资源冲突处理入口
- 审计日志记录入口

## 目标 3：统一前后端契约
至少做到：
- 一个页面只认一套状态枚举
- 一个功能只走一条正式写回路径
- 一个领域对象只认一套结构化 API 语义

---

## 四、治理优先级

## P0：项目状态统一
### 为什么优先
项目主流程是项目管理模块的中轴线。
如果状态模型不统一，后续：
- 编辑
- 状态流转
- 归档/恢复
- 合同联动
- 审计
都会持续冒刺。

### 目标
1. 明确唯一状态模型（建议 ST 状态为核心、旧字符串仅兼容）
2. 明确状态映射表
3. 所有状态变更统一走专用状态入口
4. 审计日志统一挂在状态变更链路上

### 产物建议
- `PROJECT_STATUS_MODEL.md`
- `project_status_mapping.py` / 常量表
- 前后端统一状态字典

---

## P0：模板领域模型统一
### 为什么优先
模板复用是当前项目管理模块最有复利的能力。
如果模板只是页面，不是领域对象，后续所有增强都会越来越乱。

### 目标
统一模板的：
- 元信息
- 结构（阶段/里程碑/任务/WBS/角色/周期）
- 预览数据来源
- 实例化行为
- 使用统计行为

### 产物建议
- `PROJECT_TEMPLATE_DOMAIN_MODEL.md`
- 模板预览 DTO / schema
- 模板创建项目统一 service

---

## P1：销售→项目转换入口统一
### 为什么重要
这条线已经成型，但必须防止继续“各处都能转项目”。

### 目标
1. 只保留一个正式转换入口
2. 明确转换规则与校验门槛（如 G1-G4）
3. 明确字段映射
4. 明确审计事件

### 产物建议
- `SALES_TO_PROJECT_CONVERSION_SPEC.md`
- 统一 service / command handler
- 统一审计事件名

---

## P1：资源冲突数据口径统一
### 为什么重要
资源冲突越强，越需要口径统一。

### 目标
1. 明确冲突计算维度
2. 明确冲突状态（待处理/已处理/已忽略）
3. 明确冲突处理动作与审计
4. 明确资源冲突与项目/任务的关联方式

### 产物建议
- `RESOURCE_CONFLICT_MODEL.md`
- 冲突状态枚举
- 冲突处理记录结构

---

## P2：跨模块审计与事件统一
### 为什么放 P2
值钱，但不是眼前卡主流程的点。

### 目标
统一这些事件：
- 项目创建
- 状态变更
- 模板实例化
- 资源冲突处理
- 合同转项目

### 产物建议
- `DOMAIN_EVENT_CATALOG.md`
- 审计事件命名规范

---

## 五、建议实施方式

## Phase A：梳理与冻结（1 周）
目标：先统一概念，不先大改代码。

### 做法
1. 冻结新增状态枚举/模板字段/转换入口的野生扩展
2. 写清 3 份设计文档：
   - 项目状态模型
   - 模板领域模型
   - 销售→项目转换规范
3. 标出旧兼容逻辑与正式入口

### 成果
- 让团队后续不再“想怎么接就怎么接”

---

## Phase B：关键入口收敛（1~2 周）
目标：不换框架，只收关键入口。

### 做法
1. 项目状态更新统一入口
2. 模板创建项目统一入口
3. 销售/合同转项目统一入口
4. 审计入口统一收口

### 原则
- 旧入口可以兼容一段时间
- 但新代码一律走新入口

---

## Phase C：契约统一（1 周）
目标：减少前后端语义偏差。

### 做法
1. 状态码/状态标签统一
2. 模板预览数据结构统一
3. 项目列表/详情返回结构统一
4. 冲突视图数据结构统一

---

## 六、不要现在做的事

### 1. 不要全仓重构目录
风险高，收益低。

### 2. 不要一次性把所有旧状态值都删掉
先做映射与兼容，再逐步淘汰。

### 3. 不要继续在多个地方新增“临时转换入口”
这会把 sales→project 线重新搞乱。

### 4. 不要为做治理而暂停全部业务开发
正确方式是：
- 治理关键入口
- 保持业务连续推进

---

## 七、建议的下一步动作

如果要正式开始底座治理，我建议先做：

### 第一刀
**项目状态统一**

### 第二刀
**模板领域模型统一**

### 第三刀
**销售→项目转换入口统一**

这三刀打完，项目管理、模板运营、资源冲突、销售联动都会明显更顺。

---

## 八、最终结论

当前系统：
- **不需要推倒重来**
- **但必须开始治理底座**

最优策略不是“全面重构”，而是：

# **收敛式底座治理**

即：
- 统一核心语义
- 统一关键入口
- 统一前后端契约
- 保留业务线连续性

一句话：

> **楼已经盖起来了，现在要规范水电和承重。**

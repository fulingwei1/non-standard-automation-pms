# Permission Management UX Plan

> 生成时间：2026-03-26
> 目标：将现有“工程化可用但操作繁琐”的权限管理能力，优化为业务管理员可高效使用的权限管理台

---

## 一、背景与问题定义

当前权限系统底层能力已经较为完整：
- 统一权限引擎
- 动作权限（action permissions）
- 数据范围控制（data scope）
- approvals participant / workflow visibility
- 审计脚本、unit / integration 测试、hardening 文档

但**管理台体验层明显滞后**，当前典型痛点包括：

1. **角色创建繁琐**
   - 从零创建角色步骤多
   - 缺少复制现有角色 / 套用模板能力

2. **权限分配过于工程化**
   - 管理员需要面对大量权限码
   - 缺少按业务模块组织的可视化配置方式

3. **数据范围配置抽象且难以预判效果**
   - 管理员不易理解“这个角色最终能看到哪些数据”
   - approvals 与 sales / engperf 的可见性模型差异没有被产品化呈现

4. **导入/导出与批量运营能力不足**
   - 角色导入、导出、复制、批量调整不顺手
   - 缺少导入前预校验和差异预览

5. **缺少“最终效果预览”**
   - 角色配置后，需要到前台实际试错验证
   - 权限管理仍然高度依赖测试和人工回归

### 核心判断
当前问题不是“权限系统不安全”，而是：

> **权限系统已经足够强，但权限管理台还不够好用。**

因此接下来不建议再重构权限内核，而应重点优化：

# **权限管理工作流与管理台 UX**

---

## 二、目标

### 业务目标
让业务管理员能够：
- 快速创建角色
- 快速分配权限
- 直观看到该角色的菜单/功能/数据可见性
- 通过模板、复制、导入导出完成批量运营

### 产品目标
把当前权限管理从：
- 码表驱动
- 工程视角
- 试错式配置

转为：
- 模块驱动
- 管理员视角
- 可预览、可批量、可回滚

### 技术目标
- **不推翻现有权限引擎**
- 基于当前 Sprint 1~8 已完成的能力做体验层增强
- 新增“权限元数据层 / 预览接口 / 导入校验层”

---

## 三、产品设计原则

### 原则 1：不重写权限底层
保留现有：
- permission engine
- action permission
- data scope
- participant/workflow visibility
- 审计与测试护栏

本方案只优化：
- 权限管理台
- 配置流程
- 批量运营能力
- 预览与校验能力

### 原则 2：按业务模块组织，不按权限码组织
管理员不应面对：
- `sales:quote:read`
- `approval:template:manage`
- `project_role:assign`

而应面对：
- 销售模块 → 只读 / 编辑 / 管理
- 审批模块 → 查看本人流程 / 管理模板 / 审批他人
- 工程绩效 → 查看本人 / 查看本部门 / 管理全局

### 原则 3：所有配置都应可预览
每次保存前，管理员应能看到：
- 最终菜单可见性
- 功能可见性
- 数据可见性摘要
- 被拒绝访问的关键范围

### 原则 4：模块可见性模型允许差异化
不同模块不应该被硬套同一套 scope：
- `engineer_performance` → 组织数据范围
- `sales` → 组织数据范围 + 财务特权
- `approvals` → participant / workflow visibility

前端配置层应把这些差异显式展示出来。

---

## 四、方案总览

建议将权限管理台优化拆成 4 个模块：

### 模块 A：角色模板与复制
### 模块 B：模块化权限分配
### 模块 C：数据范围与可见性配置
### 模块 D：导入导出与批量操作

---

## 五、详细方案

## 模块 A：角色模板与复制

### 目标
把“从零建角色”变成“从已有角色派生”。

### 建议功能

#### 1. 复制角色
UI 动作：
- 角色列表中新增 `复制角色`

交互流程：
1. 选择一个现有角色
2. 输入新角色名称 / 说明
3. 默认复制：
   - 权限集
   - 数据范围配置
   - 模块可见性配置
4. 跳转到编辑页微调

#### 2. 系统角色模板
建议内置模板：
- 销售
- 销售经理
- 财务
- PM
- 工程师
- 审批管理员
- HR
- 超级管理员（只读模板，不允许直接套用到普通角色）

#### 3. 保存为模板
管理员可以把现有角色保存成模板，便于跨部门复用。

### 价值
- 大幅减少重复手工配置
- 适合角色数量增多后的日常运营

### 优先级
**P0**

---

## 模块 B：模块化权限分配

### 目标
把“权限码清单”变成“模块化配置面板”。

### UI 结构建议
按模块展示：
- 用户与角色
- 项目管理
- 销售
- 审批
- 仓储
- 工程绩效
- 财务
- 系统管理

每个模块下按操作分组：
- 查看
- 创建
- 编辑
- 删除
- 审批
- 导出
- 管理

### 快捷能力
每个模块提供快速档位：
- 只读
- 编辑
- 管理
- 清空

### 后端支持建议
新增权限元数据接口，例如：
```json
{
  "module": "sales",
  "displayName": "销售",
  "groups": [
    {
      "name": "合同",
      "permissions": [
        "contract:view",
        "contract:create",
        "contract:update",
        "contract:approve",
        "contract:export"
      ]
    }
  ]
}
```

### 价值
这是最直接的体验提升点。

### 优先级
**P0**

---

## 模块 C：数据范围与可见性配置

### 目标
把最抽象、最容易配错的部分做成可理解、可预览的配置器。

### 1. 组织型模块配置器
适用于：
- sales
- engineer_performance
- 某些项目管理模块

选项：
- 本人
- 本团队
- 本部门
- 全部
- 自定义（后续）

### 2. 审批模块配置器
适用于：
- approvals

选项：
- 仅查看我发起的流程
- 仅查看我参与审批的流程
- 查看抄送给我的流程
- 查看我管理模板下的流程
- 全部可见（审批管理员）

### 3. 预览面板
右侧显示：
- 可见菜单
- 可执行操作
- 数据范围摘要
- 不可见/被拦截的关键对象

#### 示例
- 销售经理：可看本部门商机、报价、合同；不可看他部门客户明细
- 财务：可看发票、回款、付款计划；不可直接编辑客户基础信息
- 审批管理员：可看所有审批模板和实例；普通审批人仅可看自己参与流程

### 优先级
**P0**

---

## 模块 D：导入导出与批量操作

### 目标
支持权限体系的批量运营。

### 建议功能
#### 1. 角色导出
- 导出 JSON（优先）
- 导出 Excel（可选）

#### 2. 角色导入
- 下载模板
- 上传后预校验
- 权限码别名自动映射
- 导入前差异预览
- 冲突提示

#### 3. 批量分配角色
支持按：
- 部门
- 岗位
- 用户名单
进行批量赋角

### 优先级
**P1**

## 六、推荐实施顺序

1. 复制角色
2. 模块化权限分配 UI

1. 数据范围配置器
2. approvals 可见性配置器
3. 角色模板化

---

1. 导入导出
2. 批量分配
3. 角色差异对比
4. 权限变更审计记录

---

## 七、技术落地建议

### 建议新增的后端接口

#### 1. 权限元数据接口
返回模块、分组、显示名称、推荐档位。

#### 2. 角色效果预览接口
输入：
- 权限集
- 数据范围配置

输出：
- 菜单可见性
- 功能可见性
- 数据可见性摘要

#### 3. 导入校验接口
输入角色导入内容，输出：
- 无效权限码
- 权限码别名映射结果
- 冲突项
- 风险提示

---

### 建议新增的前端组件
1. `RoleTemplatePicker`
2. `PermissionMatrix`
3. `ScopeConfigurator`
4. `RoleEffectPreview`

## 八、难度评估

| 方向 | 难度 | 说明 |
|------|------|------|
| 复制角色 | 低 | 复用现有角色数据即可 |
| 模块化权限分配 UI | 中 | 需要权限元数据契约 |
| 权限效果预览 | 中 | 需要后端汇总接口 |
| 数据范围配置器 | 中 | 需按模块差异化建模 |
| 导入导出 | 中高 | 涉及模板、校验、兼容映射 |
| 批量分配 | 中 | 主要是运营能力 |
| 角色差异对比 | 中 | 偏展示层 |

### 总体判断
这是**产品/管理台体验升级**，不是权限底层重构。

---

## 九、建议的下一步

### 推荐立即做（第一批 MVP）
1. 复制角色
2. 模块化权限分配 UI
3. 权限效果预览

### 原因
- 回报最高
- 风险可控
- 不需要再动权限核心引擎
- 能直接解决你提出的“流程繁琐”问题

---

## 十、最终结论

当前权限系统的问题不是“不够强”，而是“不够好用”。

这套系统已经具备：
- 安全边界
- 动作权限
- 数据范围
- participant/workflow visibility
- 审计与测试护栏

下一阶段应从“继续补权限”切换为：

# **把强系统做成好系统**

也就是：
- 角色模板化
- 模块化配置
- 可见性预览
- 导入导出和批量运营

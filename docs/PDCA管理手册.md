# PDCA管理手册

## 1. PDCA简介

### 1.1 什么是PDCA
PDCA循环是由美国质量管理专家戴明博士提出的持续改进模型，包含四个阶段：
- **P (Plan)**: 计划 - 发现问题，分析原因，制定改善计划
- **D (Do)**: 执行 - 实施改善措施
- **C (Check)**: 检查 - 验证改善效果
- **A (Act)**: 改进 - 标准化成功经验，处理遗留问题

### 1.2 PDCA在异常管理中的应用
对于重要异常，通过PDCA循环实现：
1. 彻底解决问题
2. 防止问题重复发生
3. 持续改进流程
4. 积累管理经验

## 2. 启动PDCA的条件

### 2.1 必须启动PDCA
- 重要异常（MAJOR级别）
- 严重异常（CRITICAL级别）
- 重复异常（同类异常≥3次）
- 重大损失（影响成本≥10000元）

### 2.2 建议启动PDCA
- 典型异常（有推广价值）
- 复杂异常（涉及多部门）
- 创新改善（新方法新工艺）

## 3. Plan阶段（计划）

### 3.1 目标
明确问题，找出根本原因，制定改善计划。

### 3.2 操作步骤

#### 步骤1：问题描述
- 清晰描述异常现象
- 量化问题严重程度
- 说明影响范围

**示例**：
```
问题：A产品线设备频繁故障，导致停机
频率：近一个月发生5次
影响：每次停机2-4小时，累计损失约3万元
```

#### 步骤2：根本原因分析
使用5Why分析法或鱼骨图找出根本原因。

**5Why示例**：
```
1. 为什么设备故障？→ 轴承损坏
2. 为什么轴承损坏？→ 润滑不足
3. 为什么润滑不足？→ 润滑油添加不及时
4. 为什么不及时添加？→ 缺少定期检查机制
5. 为什么缺少机制？→ 维护计划不完善
根本原因：维护计划不完善
```

#### 步骤3：改善目标
设定SMART目标（具体、可衡量、可实现、相关、有时限）。

**示例**：
```
目标：在30天内建立设备维护计划，将故障率降低80%
衡量指标：月度故障次数从5次降至1次以内
```

#### 步骤4：改善措施
制定具体的改善措施，包括：
- 措施内容
- 负责人
- 完成期限
- 所需资源

**示例**：
```json
[
  {
    "measure": "制定设备日常点检表",
    "owner": "设备主管",
    "deadline": "2024-03-15",
    "resources": "技术文档、培训时间"
  },
  {
    "measure": "建立润滑油添加SOP",
    "owner": "维护班长",
    "deadline": "2024-03-20",
    "resources": "标准化模板"
  }
]
```

### 3.3 系统操作

#### API请求
```bash
POST /api/v1/production/exception/pdca
{
  "exception_id": 123,
  "plan_description": "设备频繁故障问题",
  "plan_root_cause": "维护计划不完善，缺少定期检查机制",
  "plan_target": "30天内将故障率降低80%",
  "plan_measures": "[{\"measure\":\"制定设备日常点检表\",...}]",
  "plan_owner_id": 5,
  "plan_deadline": "2024-04-15T00:00:00"
}
```

### 3.4 注意事项
- 原因分析要深入，避免停留在表面
- 目标要量化，便于后续检查
- 措施要具体可执行
- 责任要明确到人

## 4. Do阶段（执行）

### 4.1 目标
按计划实施改善措施，记录执行过程。

### 4.2 操作步骤

#### 步骤1：实施措施
按Plan阶段制定的措施逐项执行。

#### 步骤2：记录过程
详细记录：
- 实施了什么措施
- 使用了哪些资源
- 遇到了什么困难
- 如何解决困难

**示例**：
```
实施内容：
1. 编制了《设备日常点检表》，包含20个检查项
2. 对维护班组进行了2次培训，共12人参加
3. 建立了《润滑油添加SOP》并现场指导

使用资源：
- 技术专家2人日
- 培训费用2000元
- 标准化文档模板

遇到的困难：
- 部分老员工抵触新流程
- 点检时间与生产冲突

解决方法：
- 组织经验分享会，由老师傅示范
- 调整点检时间至换班间隙
```

### 4.3 系统操作

#### API请求
```bash
PUT /api/v1/production/exception/pdca/{id}/advance
{
  "stage": "DO",
  "do_action_taken": "编制点检表、培训、建立SOP",
  "do_resources_used": "技术专家2人日，培训费用2000元",
  "do_difficulties": "老员工抵触，时间冲突",
  "do_owner_id": 5
}
```

### 4.4 注意事项
- 严格按计划执行，不随意更改
- 如需调整，要有充分理由
- 及时记录，避免遗忘细节
- 保存证据（照片、文档、数据）

## 5. Check阶段（检查）

### 5.1 目标
验证改善措施的有效性，分析结果与目标的差距。

### 5.2 操作步骤

#### 步骤1：收集数据
收集改善前后的对比数据。

**示例**：
```
改善前：月度故障5次，平均停机3小时/次
改善后：月度故障1次，平均停机0.5小时/次
```

#### 步骤2：效果评估
评估改善有效性：
- **EFFECTIVE（有效）**：达到或超过目标
- **PARTIAL（部分有效）**：部分达到目标
- **INEFFECTIVE（无效）**：未达到目标

#### 步骤3：数据分析
使用图表展示改善效果。

**示例**：
```json
{
  "before": {
    "monthly_failures": 5,
    "avg_downtime_hours": 3,
    "cost_loss": 30000
  },
  "after": {
    "monthly_failures": 1,
    "avg_downtime_hours": 0.5,
    "cost_loss": 3000
  },
  "improvement_rate": {
    "failures": "80%",
    "downtime": "83%",
    "cost": "90%"
  }
}
```

#### 步骤4：差距分析
分析未达成目标的原因。

### 5.3 系统操作

#### API请求
```bash
PUT /api/v1/production/exception/pdca/{id}/advance
{
  "stage": "CHECK",
  "check_result": "故障率降低80%，停机时间减少83%",
  "check_effectiveness": "EFFECTIVE",
  "check_data": "{\"before\":{...},\"after\":{...}}",
  "check_gap": "个别员工执行不到位",
  "check_owner_id": 6
}
```

### 5.4 注意事项
- 数据要真实客观
- 对比要公平（同等条件）
- 分析要全面（定量+定性）
- 承认不足，不掩饰问题

## 6. Act阶段（改进）

### 6.1 目标
固化成功经验，处理遗留问题，启动新循环。

### 6.2 操作步骤

#### 步骤1：标准化
将成功措施固化为标准：
- 制度标准
- 流程标准
- 操作标准

**示例**：
```
标准化措施：
1. 将《设备日常点检表》纳入设备管理制度
2. 将点检频率固化为每班次一次
3. 将润滑油添加SOP纳入培训教材
```

#### 步骤2：横向展开
将经验推广到其他区域：
- 其他设备
- 其他产线
- 其他工厂

**示例**：
```
横向展开计划：
1. 在B、C产线推广设备点检制度
2. 对全厂设备维护班组进行培训
3. 建立跨产线经验分享机制
```

#### 步骤3：处理遗留问题
列出未解决的问题，制定后续计划。

**示例**：
```
遗留问题：
1. 部分老旧设备点检表不适用
2. 润滑油消耗量偏高

下一轮PDCA：
- 针对老旧设备重新设计点检表
- 分析润滑油消耗原因
```

#### 步骤4：总结经验
记录整个PDCA过程的经验教训。

### 6.3 系统操作

#### API请求
```bash
PUT /api/v1/production/exception/pdca/{id}/advance
{
  "stage": "ACT",
  "act_standardization": "纳入设备管理制度，固化点检频率",
  "act_horizontal_deployment": "推广至B、C产线，全厂培训",
  "act_remaining_issues": "老旧设备点检表不适用",
  "act_next_cycle": "重新设计老旧设备点检表",
  "act_owner_id": 5
}
```

#### 完成PDCA
```bash
PUT /api/v1/production/exception/pdca/{id}/advance
{
  "stage": "COMPLETED",
  "summary": "通过PDCA循环，成功降低设备故障率80%",
  "lessons_learned": "定期点检比事后维修更有效；标准化是持续改进的关键"
}
```

### 6.4 注意事项
- 标准化要落地，不能流于形式
- 横向展开要因地制宜
- 遗留问题要跟踪，不能不了了之
- 经验总结要深刻，提炼规律

## 7. PDCA循环

### 7.1 螺旋上升
PDCA不是一次性活动，而是循环往复：

```
第一轮PDCA：解决主要问题
    ↓
第二轮PDCA：优化解决方案
    ↓
第三轮PDCA：横向展开推广
    ↓
持续改进...
```

### 7.2 大环套小环
- **大环**：年度质量改善PDCA
- **中环**：季度流程优化PDCA
- **小环**：月度异常处理PDCA

## 8. 常见问题

### Q1: Plan阶段花了很长时间，正常吗？
A: 正常。"磨刀不误砍柴工"，Plan阶段分析越充分，后续执行越顺利。通常Plan阶段占整个PDCA周期的30-40%。

### Q2: Check阶段发现效果不理想怎么办？
A: 分两种情况：
1. 措施本身有问题 → 退回Plan阶段，重新分析原因
2. 执行不到位 → 退回Do阶段，加强执行

### Q3: 可以跳过某个阶段吗？
A: 不可以。系统强制按PLAN→DO→CHECK→ACT顺序推进，确保PDCA完整性。

### Q4: 一个异常可以启动多个PDCA吗？
A: 可以。复杂问题可能需要多轮PDCA，每轮聚焦不同方面。

### Q5: PDCA需要多长时间完成？
A: 根据问题复杂度：
- 简单问题：1-2周
- 一般问题：1个月
- 复杂问题：2-3个月

## 9. 最佳实践

### 9.1 成功案例：设备故障率降低
- **背景**：某产线设备月度故障5次
- **Plan**：分析根因为维护不到位，制定点检计划
- **Do**：编制点检表，培训员工，建立SOP
- **Check**：故障率降低80%，达成目标
- **Act**：标准化推广至全厂，故障率持续下降

### 9.2 失败教训：表面化改善
- **问题**：改善措施停留在表面，未找到根本原因
- **后果**：问题反复发生，PDCA流于形式
- **教训**：一定要用5Why等工具深挖根因
